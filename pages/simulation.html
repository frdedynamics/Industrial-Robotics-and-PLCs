Simulation

Why simulate?

Before a robot cell exists, many decisions already have to be made. Can the robot reach all required poses? Is there enough space for the tool and workpiece? Will the cell meet the required cycle time? How should the robot, conveyors, sensors, and PLC coordinate? Building the cell first and answering these questions afterwards is slow, expensive, and difficult to change.

A simulation lets you investigate such questions using an executable model of the planned or existing system. You can change the model, run an experiment, observe the result, and compare alternatives without immediately changing the real equipment. This makes simulation useful for robot motion, complete cells, production lines, training, and communication with customers and operators. Here, executable means that the model changes over time: components move, signals change, and the modeled equipment responds according to defined behavior. A static 3D drawing can be part of a simulation model, but it is not a simulation by itself.

Consider a palletizing cell in which a robot picks boxes from a conveyor and places them on pallets. A simulation could help select the robot and gripper, develop the layout, check reachability and collisions, estimate cycle time and production capacity, and test how the PLC sequence responds to missing boxes, full pallets, or stopped equipment. The same model can also make the proposed cell easier to discuss with the customer and the people who will operate it.

These are not all the same simulation problem. A rough layout may be sufficient for checking floor space, while cycle-time prediction or testing connected control software needs much more detailed behavior. The purpose of the simulation therefore determines what must be included in the model.

The questions above are not only technical checks. Together, they can provide an early proof of concept before investing in the robot, tooling, fixtures, sensors, safety equipment, engineering, and installation. The simulation helps show whether a concept deserves further development and which uncertain parts still need more investigation.

These technical results can support the business case. For example, simulated cycle time and production capacity can be combined with expected production volume, labor requirements, integration costs, and operating costs to estimate payback and compare alternative concepts. Simulation does not calculate profitability by itself, but it can replace important guesses with better evidence while changes are still inexpensive.

This is where simulation connects usefully to Lean Robotics. The methodology recommends comparing the current and proposed tasks, calculating return on investment, and de-risking critical assumptions before finalizing the robotic cell concept (Bouchard 2017). Simulation is one practical tool for supporting those steps; it is not itself a Lean Robotics principle.

In this course, we use Visual Components Premium OLP 5.1 as our main environment for building robot cells, simulating production behavior, developing robot operations, and connecting simulation with PLC programming. It is one example of a simulation platform, not the only possible choice.

Important

A simulation is not automatically a correct prediction of reality. It answers a question within the assumptions and limitations of its model. A convincing 3D animation can still give a wrong result.

A model built for a purpose

A model is a deliberate simplification. It contains the parts of reality that matter for the current question and leaves other details out. A useful model is therefore not necessarily the most detailed model; it is detailed enough to support the decision you need to make.

You have probably encountered George Box’s well-known reminder: “Essentially, all models are wrong, but some are useful” (Box and Draper 1987, 424). Here, wrong means incomplete or simplified, not worthless.

The model boundary separates what is represented inside the simulation from what is treated as an external input or left out. A robot-only model has a narrow boundary. A production-line model may include several machines, operators, material flow, and control systems.

Building a working simulation has an important side effect: it forces you to make details explicit. You must decide where a sensor is located, when a part appears, which signal starts a motion, how long an operation takes, and what happens next. Questions that might remain hidden in a drawing or meeting often become visible because the simulation cannot run until some behavior has been defined.

This can also create false confidence. The simulation only forces you to define what is represented in the model. A sensor may work perfectly by default, a gripper may never drop a part, and communication may have no delay. Once the virtual cell runs smoothly, it is easy to forget which real-world details are still missing. Treat a successful run as evidence about the modeled system, not as proof that the physical cell is complete.

Question Information the model needs
Can the robot reach the task? Correct robot kinematics, tool, workpiece poses, frames, and joint limits
Can the motion collide with something? Relevant collision geometry and the actual robot paths
What is the robot cycle time? Controller behavior, speeds, accelerations, blending, tool actions, and process delays
Can the cell meet its production target? Robot and machine cycle times, part arrivals, buffers, failures, and control logic
Does the PLC sequence work? Sensors, actuators, I/O mapping, state changes, timing, and fault scenarios
Is the layout suitable for the people using it? Access routes and the tasks performed during production, teaching, recovery, and maintenance

For example, a simplified box is often sufficient when checking whether a workpiece collides with a fence. The same box may be inadequate when checking whether a gripper finger fits into a narrow opening. Adding detail that cannot change the answer only makes the model slower to build and harder to maintain.

Model fidelity and the simulation-to-reality gap

Model fidelity describes how closely the relevant behavior of the model matches the real system. Fidelity is not one overall percentage. A model can have accurate robot geometry but unrealistic sensor timing, or realistic material flow but only approximate robot motion.

Different levels of fidelity within one model are not only acceptable but often desirable. Use high fidelity where an error could change the decision, and use simpler representations where additional detail would not affect the result. This avoids spending unnecessary time, data preparation, maintenance effort, and computing resources while preserving accuracy where it matters.

The difference between simulated and real behavior is often called the simulation-to-reality gap, or simply the sim-to-real gap. In industrial robotics, common causes are inaccurate geometry, frames, payload data, or robot configurations; idealized sensors, gripping, contact, and communication; differences between generic simulated motion and the real controller; omitted process variation and recovery situations; and physical changes that were not added to the model.

The gap cannot be removed by making the graphics more realistic. It is reduced by using trustworthy input data, calibrating the real cell, comparing important simulation results with independent calculations or measurements, and updating the model when the installation changes.

Two checks are useful. Verification asks whether the model has been implemented as intended. For example, are signals connected to the correct simulated sensors, and does the gripper logic follow the specified sequence?

Validation asks whether the model represents the real system well enough for its intended use. For example, do measured robot cycle times agree closely enough with the simulated times for a production-capacity decision?

Validation is always connected to a purpose. A model validated for explaining a cell concept to a customer is not automatically accurate enough for predicting throughput or transferring a robot program.

The evidence behind a simulation should remain traceable. Record the sources and versions of important input data, the assumptions and limitations, the scenarios that were tested, and the model version used to produce the result. Relevant scenarios may include normal production, product variants, waiting conditions, and selected faults rather than only one ideal cycle.

Levels of simulation

Simulation models can focus on different levels of an automation system. A robot-level model answers detailed questions about the manipulator and its motion, while a cell or production-level model describes how equipment, products, control systems, and people interact. These are complementary views, not project phases that must be completed in a fixed order. Modern simulation tools such as Visual Components naturally combine both levels in one model, so robot motion and the surrounding production flow can be studied together.

When an existing manual process is being automated, the current task and layout provide the natural starting point, as described later in the Lean Robotics workflow. For a completely new production task, there is no manual cell to map. The starting point is instead the required input, output, process, production target, and constraints. In both cases, the simulation boundary may eventually include the tool, fixtures, workpieces, conveyors, machines, sensors, operators, PLC, and other robots.

Robot-level simulation

At robot level, simulation is closely connected to the concepts from frames and poses, kinematics, path planning, and motion control. Path planning determines a feasible motion between poses, while motion control governs how the robot axes follow the commanded motion. At this level, simulation is mainly used to check reachability and collisions, identify difficult joint configurations or singularities, and estimate cycle time. For process tasks, the simulated path also helps verify that the tool keeps the required orientation relative to the workpiece.

In Visual Components, this corresponds to teaching positions and building robot programs from routines and statements, with the appropriate tool and base frames. Its robot-simulation tools can then be used to inspect reachability, collisions, and cycle time. In Premium OLP, Path Check can inspect the interpolated motion between program statements; path-solving tools can also be used where the corresponding add-on is available. The Visual Components robot-simulation learning path introduces these software concepts.

For the palletizing example, the robot can reach the top of a new pallet quite easily but may struggle with the lowest boxes at the far side. A simulation lets you test the robot position, pallet orientation, tool length, and approach direction before fixing the layout.

Cell and production-flow simulation

At cell or production-line level, the main question is often not one robot pose but how resources interact over time. A box arrives, a sensor changes state, the PLC starts the robot, a machine finishes, or a buffer becomes full. These changes are commonly represented as events in a discrete-event simulation.

This broader model can reveal production rate, queues, buffers, bottlenecks, and resource utilization. It can also show how product variants, failures, material shortages, or coordination between processes affect the complete system.

In Visual Components, the closest counterpart is Process Modeling. A product is an item moving through production, a process defines what happens at a station, and a flow defines the sequence of processes that the product follows. The Process Flow editor is used to define this sequence, while transport links and controllers represent movement between processes. The Introduction to Process Modeling uses this vocabulary to build production-flow simulations.

A robot path that is two seconds faster may have no effect on total production if the robot is usually waiting for the upstream machine. Conversely, a small buffer may increase production more than a faster robot. Whole-cell simulation helps reveal these system-level effects.

Simulation through a Lean Robotics project

Lean Robotics organizes robotic-cell deployment into four phases: Prepare, Design, Integrate, and Operate (Bouchard 2017). Simulation is not an additional phase. It is a tool that can support decisions and reduce uncertainty throughout this workflow. Figure 1 keeps the same four-phase structure.

The Design phase in Lean Robotics moves from understanding the current manual task to defining and comparing a robotic alternative before the cell concept is finalized. This is where simulation can provide an early proof of concept and help de-risk the minimum viable robotic cell: a deliberately simple first cell that can deliver the required production value. The model can then be developed further during integration and corrected using observations from operation.

Lean Robotics phase How simulation can support the project
Prepare Define the task, desired output, production target, and questions the simulation must answer. Gather the data needed to model the current task.
Design Represent the current process and proposed robotic process, compare layouts and equipment, and investigate reachability, material flow, cycle time, capacity, and important uncertainties before committing to a concept.
Integrate Develop robot programs and detailed cell behavior, connect representative control software to the simulated cell, and test sequences, signals, product variants, and selected faults before installation.
Operate Compare predicted and measured performance, update the model when the cell changes, and test possible improvements before modifying production equipment.

Lean Robotics provides the deployment method; Visual Components provides models and tests that support parts of that method. The connected controller testing described in Section 6 fits mainly within the Integrate phase, but it depends on the cell model and decisions developed during Design.

Simulation is useful for discussing safety early because layouts, access, tasks, and abnormal situations can be made visible and changed while the design is still flexible. It does not validate the safety of the physical installation. The limits of simulation for safety work are discussed further in Simulation as a safety tool.

Offline robot programming

Simulation and offline programming are related but not identical. Simulation is the broader activity of experimenting with a model. Offline programming (OLP) uses a simulation environment to create or modify a robot task program away from the physical robot. The complete OLP workflow, including postprocessing and transfer, is covered on the robot programming page.

Figure 2 shows a typical OLP environment. The 3D model provides the context for tools, frames, workpieces, targets, paths, collisions, and robot configurations.

A visually correct simulation is not enough for program transfer. The real robot must agree with the model about the TCP, bases, fixture and workpiece positions, robot configuration, controller options, and program semantics. Calibration connects the coordinate systems of the virtual and physical cells.

Generic models and virtual controllers

A general simulation tool can calculate robot motion from a kinematic model and then translate the result through a postprocessor. This is flexible and supports many robot brands, but details such as path blending, motion timing, and special instructions may differ from the real controller.

A virtual controller runs the robot vendor’s controller software, or a closely corresponding software version, on a computer. It can provide better agreement for program syntax, configuration, motion generation, I/O behavior, and cycle time. For example, ABB describes RobotStudio as using an ABB Virtual Controller, and KUKA combines KUKA.Sim with controller and connectivity options for programming and testing a simulated cell.

Even a virtual controller does not make the complete cell real. Contact, gripping, process behavior, external equipment, communication, and mechanical variation still depend on the surrounding model.

PLC integration and virtual commissioning

Offline robot programming tests the robot program. A working cell also needs the PLC and robot to coordinate with sensors, grippers, conveyors, machines, and operator commands. These control programs can be developed at the same time as the virtual cell.

In this course, we develop the PLC program in CODESYS and run it with CODESYS Control Win SL, a software-based PLC runtime, or SoftPLC, for Windows. Selected PLC variables are exchanged through OPC UA, with Visual Components acting as the client and mapping the variables to signals of the simulated equipment. We can therefore develop the PLC program and virtual cell together while each executes in its own environment. The Visual Components Academy lesson on connecting a remote OPC UA server demonstrates the complete setup, including the CODESYS steps.

In a connected simulation, the signal loop works like this:

  1. A simulated sensor or device produces an input signal.
  2. The PLC or controller executes the actual control logic.
  3. Its outputs command simulated actuators or robot actions.
  4. The model changes state and returns new sensor signals.

For the palletizing example, the virtual conveyor can activate part_at_pick. The PLC then checks that the robot is ready and permits the pick. The robot reports pick_complete, and the PLC releases the next box. Faults such as a missing box, failed grip, full pallet, or lost ready signal can be tested before all physical equipment is available. Such an agreed sequence of signals between controllers is often called a handshake.

Commissioning is the work of bringing the installed cell into operation and checking that its equipment and control programs behave as intended. The core idea of virtual commissioning is straightforward: perform part of this testing against a sufficiently representative simulation before the physical cell is ready. It is a structured use of simulation, not a separate kind of automation. What distinguishes it from simply watching a simulation is that the control program and interfaces being tested represent the planned real implementation.

When both the plant model and controller run as software, the setup is commonly called software-in-the-loop. Our CODESYS Control Win SL and Visual Components setup is an example. If a physical PLC or robot controller is connected to the model, it is commonly called hardware-in-the-loop. The exact boundary depends on the available software, interfaces, and project goals.

This approach allows robot programmers, PLC programmers, and mechanical designers to work in parallel. Errors in the I/O list, sequence, interlocks, timeouts, reset behavior, and product handling can be found while the mechanical cell is still being assembled. An interlock is a condition that blocks an action until the required state is true, for example preventing the conveyor from moving while the robot is picking a box. The official Visual Components virtual commissioning learning path places this controller connection in the broader context of component behavior, PLC connectivity, and robot-controller environments.

Testing against the virtual cell reduces the amount of control debugging that must happen for the first time on the physical cell. It does not remove physical commissioning. Wiring, sensor placement, network behavior, mechanical tolerances, calibration, stopping performance, process quality, and safety functions still have to be checked on the finished installation.

Simulation and digital twins

The term digital twin is often used loosely, but not every CAD model or simulation is a digital twin. A practical distinction is:

Digital representation What it contains Connection to the real system
3D model Geometry and component structure Usually none
Simulation model Executable behavior for experiments and predictions May be completely offline
Digital twin A digital representation of a particular physical asset or system that is updated using operational data An ongoing data connection keeps relevant state aligned with the physical counterpart and may also support feedback to it

Definitions vary between organizations and applications. The important additional idea is the continuing connection to a particular physical asset or process. The Norwegian research organization SINTEF describes digital twins as virtual representations built from historical and real-time data, and emphasizes two-way communication where the twin is used to support decisions or actions in the physical system (SINTEF: Data-Driven Digital Twins).

For the palletizing cell, an offline simulation might use an assumed arrival rate and robot cycle time. A digital-twin implementation could receive actual PLC states, production counts, robot positions, alarms, and measured cycle times. The connected model could then monitor production, reveal differences between expected and measured behavior, test changes to schedules or product mix, help diagnose repeated stops, and support maintenance planning.

For predictive maintenance, operational measurements such as motor current, temperature, vibration, cycle count, or fault history are combined with a model or data analysis to estimate deterioration and future maintenance needs. A 3D visualization alone cannot make such a prediction. The data must be relevant, reliable, and connected to a validated health model.

Maintaining a digital twin is an engineering task of its own. Model versions, controller changes, replaced equipment, sensor quality, data ownership, connectivity, and cybersecurity must be managed throughout the cell’s life. A digital twin that no longer matches its physical counterpart can support worse decisions than a simpler but well-maintained simulation.

Choosing a simulation tool

No single tool is best for every question. Some tools focus on multi-vendor factory layouts and material flow, some on detailed robot-controller behavior, and others on PLC logic or physics. The required interfaces and the software supported by the equipment vendor are often as important as the 3D functions.

Tool category and examples Strong fit
Visual Components Multi-vendor robot cells and production systems, layout and material-flow simulation, OLP, and PLC or robot-controller connectivity.
RoboDK Multi-vendor robot simulation and OLP, especially robot paths, machining and process applications, calibration, and program generation.
Vendor-specific tools such as ABB RobotStudio, KUKA.Sim, and FANUC ROBOGUIDE Detailed work with one vendor’s robots, program format, controller behavior, configuration, and optional application packages.
Factory I/O Learning and testing PLC and automation logic with configurable virtual factory scenes and connections to physical or simulated PLCs. It is not primarily a detailed robot OLP system.

Choose the tool from the question you need to answer, the required robot and PLC interfaces, and the fidelity that can be achieved and validated. Also consider who will build and maintain the model; the most capable software is not useful if it cannot be supported throughout the project.

Exercises

  1. Define three useful simulation questions for the palletizing example. For each question, identify the model boundary, required input data, useful output, and one assumption that could make the result wrong.
  2. In Visual Components, create two feasible layouts for the same robot, conveyor, and palletizing task. Compare reachability, collisions, robot configurations, floor-space use, and cycle time. Recommend one layout and state which conclusions still require physical checks.
  3. Introduce one deliberate model error, such as a wrong TCP, displaced pallet frame, idealized sensor, or missing process delay. Run the simulation before and after correcting it. Explain which conclusions were affected and why the error was or was not visually obvious.
  4. Define a PLC–robot handshake for the palletizing cell. Include at least part_at_pick, robot_ready, pick_complete, and a fault or reset signal. Test the normal sequence and two abnormal situations in simulation.
  5. Create or inspect an offline robot program. List the information that must match between the simulated and real cell before transferring it. Separate what can be checked in simulation from what must be calibrated or tested on the physical installation.
  6. Classify each of the following as a 3D model, simulation, virtual commissioning setup, or digital twin: a CAD layout, an animated robot cycle, a PLC controlling a simulated conveyor, and a live model receiving production data from an installed cell. Explain what additional connection or behavior moves each example to the next category.
  7. Use the same simulated cell for the risk-assessment exercise on the robot safety page. Identify which observations are supported by the model and which safety conclusions require information or tests from the physical installation.
